# Docs Strategy

* **言語**: **日本語必須**。
* **配置**: `docs/` 配下。役割分割に応じて整理。docs/配下に直接ドキュメントを作成しないこと。常にどのようなサブディレクトリに配置するべきか考えること。

  * 仕様: `docs/spec/`（例: `docs/spec/models/`, `docs/spec/utils/`, `docs/spec/io/`）
  * 設計: `docs/design/`（例: `docs/design/models/`, `docs/design/utils/`, `docs/design/io/`, `docs/design/adr/`）

---

### 仕様（Spec）ドキュメント
* **目的**: 実装完了後にコードを見ずにその役割や使用を把握するため。また、プロジェクトが進行するにつれて、内容のわからないコードが増えるのを防ぐため。
* **原則**: 実装仕様を疑似コードベースで特にワークフローを意識して記述。テンソルなどのデータを扱う場合はその形状の繊維を明確にワークフローの中で記述する。

### 設計（Design）ドキュメント

* **目的**: 実装前に方針を固めるため。厳密な設計を作り、それに基づいた実装を行うことで観測性を高めるため。
* **原則**: 疑似コードベースで記述する。モジュラーな設計を意識し、１つのファイルに責任が集中することを避ける。既存のディレクトリ構成を確認し、自然な拡張になるように心がける。
